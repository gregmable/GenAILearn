Orchestration & Automation

GenAI Agent – The AI Co-worker

What is a GenAI Agent?

Think of a GenAI Agent as an intelligent AI co-worker embedded in your Pega application. It's like having a smart assistant that can:

  • Interact with users in plain language
  • Take actions on the system (trigger processes, retrieve data, update cases)
  • Guide workflows with intelligence

Unlike a simple FAQ bot, a Pega GenAI Agent can actually DO things: it can trigger automation, retrieve data, start sub-cases, or update the case—all based on natural-language requests or predefined triggers.

Why Do We Have Agents?

The GenAI Agent exists to speed up work and guide users. By automating routine steps and handling queries, it lets human workers focus on higher-value decisions.

  • Real-time support: Ask it questions or have it handle a subtask immediately
  • Reduces wait time: Instant answers and action
  • Conversational experience: Provides guidance within the context of a case

For developers: you configure what tasks the Agent can do and what knowledge it has access to. For end-users, the Agent is a helpful part of the app, often via a chat interface.

Real-World Scenario: Agent in Action

Adjuster: "Summarize this claim."

GenAI Agent: "Sure. The customer reports a rear-end collision, moderate vehicle damage, no injuries, and is requesting a claim for repairs."

Adjuster: "What's the next recommended step?"

GenAI Agent: "Next, verify coverage and estimate repair cost. I can start a sub-process for damage appraisal if you'd like."

The Agent combines conversation with automated action—making it feel like talking to a knowledgeable colleague who can take immediate action.

GenAI Tool – The Agent's Toolbox

What is a GenAI Tool?

While the GenAI Agent is the "AI coworker", Tools are the specific skills or actions that coworker can perform. A GenAI Tool is essentially a rule that defines a particular capability for an Agent.

Analogy: If the Agent is a smartphone, Tools are the apps installed on it. Each Tool adds a new skill—a specialized function the Agent can execute.

Two key parts of a Tool:

  • What action: Run an automation, start a case, fetch data, etc.
  • When to use it: Example user prompts that trigger this action

Why Do We Have Tools?

Tools are about modularity and control. They let developers equip the GenAI Agent with specific, well-defined abilities rather than giving it free rein.

  • Predictable behavior: Agent only does pre-approved things
  • Security & compliance: Each Tool acts as a guardrail
  • Expandable: Add new capabilities over time by creating new Tools
  • Reusable: Same Tool can work across different Agents
  • Maintainable: Test and update each tool independently

Real-World Scenario: Tools at Work

Tool Example 1 — Policy Lookup Tool:

Adjuster: "What's the customer's coverage limit?"

Agent: (using Policy Lookup Tool) "The policy's property damage coverage limit is $20,000 and the deductible is $500."

Tool Example 2 — Damage Appraisal Tool:

Adjuster: "Go ahead and initiate the damage appraisal."

Agent: (using Damage Appraisal Tool) "Done. I've started a damage appraisal case and assigned it to the appropriate team."

Each Tool is a bite-sized piece of automation the Agent can perform on demand. They work like building blocks: the Agent picks the right tool to use based on what the user needs.

Coordination Model

Pega-Specific Orchestration Concerns

  • Expose capabilities as governed tools instead of hard-coded agent behaviors.
  • Keep authorization boundaries explicit for every model, tool, and agent action.
  • Use AgentX to coordinate specialized agents rather than overloading a single assistant.
  • Design escalation rules and human review paths up front for ambiguous outcomes.
Topic Guide

Protocols Are Only Part of the Story

MCP and A2A create integration consistency, but the harder design problem is still operational governance: which tools are safe to expose, when agents may act autonomously, and what must be escalated to a person.

Pega's value here is in combining protocol support with workflow context and case controls so tool usage stays observable and bounded by enterprise policy.
